# frida-analykit AI Agent Guide

## 目的

本文件用于约束 AI agent 在 `frida-analykit/` 项目中的开发、排错、重构与测试组织方式。

目标不是单纯“把功能写出来”，而是让改动具备以下特征：

- 可验证
- 可回退
- 低耦合
- 易于被后续 AI 和人类继续维护

本文件会持续补充。若某个场景暂未覆盖，默认遵循以下原则：

- 优先最小必要改动
- 优先可解释而不是堆补丁
- 优先清晰结构而不是临时凑合

## 基本原则

- 排错时，先区分“猜测性修复”和“已确认修复”，不要混淆。
- 修复真实问题时，优先保留能够支撑判断的证据，例如复现路径、日志、测试、调用链。
- 不要为了“可能以后有用”而引入长期复杂度。
- 临时排查代码在完成定位后，应明确决定保留还是删除，不要无期限残留。
- 代码组织应同时服务于人类维护和大模型上下文理解。

## 排查错误时的修复规则

### 1. 区分猜测性修复与已确认修复

- 猜测性修复：当前没有充分证据，只是基于怀疑做的试探性改动。
- 已确认修复：有明确证据支持，例如复现结果、日志、测试失败、调用链分析、已定位的代码行为。
- AI agent 必须清楚当前改动属于哪一种，不要把猜测性修复伪装成最终结论。

### 2. 猜测性修复必须保持可回退

- 猜测性修复应尽量小、尽量局部、尽量减少连带影响。
- 不要为一个尚未确认的问题同时叠加多层“防御式补丁”。
- 如果某个猜测性修复落地后，问题仍然存在，且后续确认根因不在这里，应评估并回退这部分猜测性代码。
- 不要让“曾经尝试过”的修复代码长期留在主干中，除非它已经被证明具有独立价值。

### 3. 修复真实问题时，非显而易见的改动必须加注释

- 如果最终修复属于非常识、非显而易见、依赖上下文知识、兼容性细节、时序约束，或某种框架/平台特殊行为，则必须在代码附近添加简短注释。
- 注释应解释“为什么必须这样改”以及“它在防止什么问题”，而不是重复代码字面含义。
- 如果修复本身是常规且显而易见的，则不必为了注释而注释。

### 4. 排错时优先简化代码，而不是继续加复杂度

- 排查问题时，优先减少分支、收敛状态、隔离变量、缩小影响范围。
- 如果能够通过删减冗余逻辑、拆小职责、收敛接口来定位问题，就不要优先通过增加更多条件分支来掩盖症状。
- 为排查临时加入的日志、调试分支、保护逻辑，在问题确认后应明确清理或保留的理由。

## 代码耦合与拆分规则

### 1. 文件过大或职责过多时，主动拆分

- 如果一个文件代码行数过多，或者一次性承载了过多不同职责，应主动考虑拆分。
- 如果为了完成一个改动，需要在同一文件中反复跳转多个不相关区域，通常说明该文件已经过于耦合。
- 不要长期把入口编排、业务逻辑、数据结构、工具函数、平台适配、测试辅助混在同一个文件里。

### 2. 拆分应按逻辑边界进行

- 拆分时应围绕明确的逻辑边界，而不是机械地按行数切块。
- 优先按职责或领域拆分，例如协议与消息定义、会话与生命周期管理、CLI 与配置处理、平台兼容层、通用工具与辅助函数。
- 目标是让每个文件在单次阅读中就可以理解其主要职责。

### 3. 文件数量增长后，使用目录或包进行组织

- 当相关文件变多时，不要继续平铺堆放。
- 应按功能域使用目录或包进行分组。
- 最终通过一个主入口文件或包入口文件对这些模块进行组织、编排和导出。
- 入口文件应尽量保持薄，主要负责装配、调度和对外暴露接口，不承载大量细节实现。

### 4. 测试代码也必须拆分

- 单元测试、集成测试、设备测试都不应发展成“大而全”的单文件。
- 测试应按模块、行为、场景或错误类型拆分，避免一个测试文件覆盖过多不相关内容。
- 公共 fixture、helper、构造逻辑、断言辅助可以提取到共享测试模块中。
- 这样做不仅是为了可维护性，也是为了降低大模型上下文压力，让 AI 更容易准确理解测试意图和项目结构。

### 5. 结构优化应服务于 AI 友好性

- 更短的文件、更单一的职责、更清晰的目录结构，会直接提升 AI agent 的理解效率与修改准确率。
- 当不确定是否应该拆分时，优先选择更容易阅读、搜索、定位、替换和复用的组织方式。
- 拆分后应保持命名清晰、依赖方向稳定、对外接口尽量简单。

## 依赖显式化与入口设计规则

### 1. 入口文件必须薄，且只暴露真实公共入口

- 包根 `__init__`、入口模块、CLI 根模块主要负责注册、装配、导出，不承担内部依赖转发层。
- 内部模块应直接从真实定义位置导入依赖，不要通过包根反查内部对象。
- 如果用户没有明确要求内部兼容，不要为了兼容旧的内部导入或 monkeypatch 方式而保留额外中转层。
- 反模式关键词：
  - `_api()`
  - `import_module("frida_analykit.xxx")` 后再 `getattr`
  - 把包根当内部 patch 汇聚点

### 2. 类的依赖必须在类自身可见

- 单独打开一个类时，应该能从其字段、构造器参数、类型注解、imports 理解主要依赖。
- 不允许一个类依赖“宿主类碰巧提供的属性或方法”才能成立。
- 如果组件需要共享运行时依赖，应通过 `runtime/context` 对象或显式字段传入，而不是隐式从宿主 `self` 上取值。
- 反模式关键词：
  - `self._registry_path` 但类内没有该属性来源
  - `self._ensure_remote_config()` 但类内看不到定义或注入
  - 当前类必须靠 mixin 宿主补全契约

### 3. 默认优先组合，不默认使用隐式 mixin

- 当目标是拆分大类或大文件时，优先使用 `manager/facade + component` 的显式组合结构。
- 推荐优先考虑 `runtime/context`、`facade/manager`、`component/store/client/controller` 这类清晰命名。
- mixin 只适用于依赖极少、契约非常稳定、且单看类本身仍然清楚的场景；否则不要使用。
- 组件间共享校验逻辑优先写成纯函数、轻量 helper 或专门组件，不挂在宿主 manager 私有方法上供别的组件隐式调用。
- 反模式关键词：
  - 一个 manager 继承多个 `*Mixin`
  - mixin 之间通过宿主私有方法串联
  - manager 退化成隐藏依赖袋

### 4. 不要为了内部兼容而保留魔法中转层

- 如果用户没有明确要求内部兼容，就不要为了延续旧 monkeypatch 入口、旧 re-export、旧 helper 名字而保留额外中间层。
- 重构时优先保证公共 API 稳定，内部依赖方向可以收缩、清理、重定向。
- 应明确区分：
  - `公共 API` 可以为兼容性保留
  - `内部实现形态` 不默认保留兼容
- 反模式关键词：
  - 为了兼容旧内部引用保留包根 re-export
  - 明知当前版本可调整，仍继续套动态壳

### 5. 测试必须 patch 显式拥有者，而不是碰巧可达的中转面

- 测试 monkeypatch 应命中真实归属模块、真实组件对象、真实拥有者方法。
- 不应把包根、manager 私有 helper、mixin 泄漏面当作长期测试入口。
- 如果重构后依赖拥有者发生变化，应同步重写测试 patch 目标，不要为了兼容旧测试继续保留错误的内部结构。
- 反模式关键词：
  - patch `frida_analykit.cli` 包根内部 helper
  - patch manager 上仅为 mixin 服务的私有方法
  - 测试依赖碰巧存在的私有结构

## Python 类型注解规则

- Python 属于弱类型语言；如果类型信息长期缺失，后续维护、重构和阅读成本会快速上升。
- 因此在本项目中，AI agent 编写或修改 Python 代码时，应主动补全和维护类型注解，而不是把类型信息留给后续猜测。
- 新增或修改 Python 代码时，优先为函数参数、返回值、类字段、实例属性、配置结构、字典字段等补齐清晰的类型标注。
- 仅仅让代码“能跑”不够；类型信息也应足够清楚，便于后续 AI 和人类理解数据结构与调用约束。
- 如果某个对象结构固定，不要长期使用语义模糊的裸 `dict`、裸 `list` 或大面积 `Any` 应付过去，应优先考虑更明确的表达方式。
- 如果确实无法给出精确类型，也应尽量给出比 `Any` 更窄的近似类型，并在必要时补充注释说明限制原因。
- 对于历史代码，如果本次改动已经触达某段 Python 逻辑，且补类型注解成本合理，应顺手改善局部类型信息。
- 类型注解的目标不是形式化打卡，而是降低理解门槛、减少误用概率、提升重构安全性。
- 注入式依赖必须有明确类型，不要把模块对象、运行时对象、`subprocess` / `urlopen` 一类依赖塞进无类型字段。
- 如果需要抽象依赖面，只允许使用“小而明确的 `Protocol`”，并定义在使用侧；不要用大而泛的接口占位。
- 不要用“返回一个不透明对象或模块，再动态调用其方法”的方式规避类型表达。

## 子系统设计规范规则

- 根目录 `AGENTS.MD` 是本项目所有端共用的公共规范底线。
- 在此基础上，本项目额外维护两份子系统级 `DESIGN_SPEC.MD`：
  - `src/frida_analykit/DESIGN_SPEC.MD`
  - `packages/frida-analykit-agent/DESIGN_SPEC.MD`
- 当改动 Python 侧代码、测试、模块边界或接口设计时，必须同时参考 `src/frida_analykit/DESIGN_SPEC.MD`。
- 当改动 agent / TypeScript 侧代码、测试、导出边界或 capability 设计时，必须同时参考 `packages/frida-analykit-agent/DESIGN_SPEC.MD`。
- 如果改动跨越 Python 与 agent 的协议边界，例如 RPC 协议、REPL 句柄、导出形态、打包导入边界或组合测试工作流，应同时对照两份 `DESIGN_SPEC.MD`。
- 子系统 `DESIGN_SPEC.MD` 只能细化该端当前风格，不能放宽或绕过本文件已有规则。
- 如果本次改动已经改变了某一侧当前的设计风格、模块组织、测试样式、导出边界或接口习惯，应同步更新对应 `DESIGN_SPEC.MD`，避免规范长期落后于代码。

## README 文档维护规则

- 本项目维护四份对外 README，且它们面向的发布面不同：
- 根 `README.md`：面向整体用户，描述整个 `frida-analykit` 项目的主线使用方式、CLI 接口、环境准备、接入流程和整体发布物。
- 根 `README_EN.md`：是根 `README.md` 的严格英文翻译版本，不是独立事实来源。
- `packages/frida-analykit-agent/README.md`：面向 npmjs 上的 `@zsa233/frida-analykit-agent` 用户，只描述这个包自身的定位、安装、导出能力、使用方式和与 agent 工作区直接相关的细节。
- `packages/frida-analykit-agent/README_EN.md`：是包文档 `packages/frida-analykit-agent/README.md` 的严格英文翻译版本，不是独立事实来源。
- `PRE_README.MD` 是上述 README 的共同中间来源文档，也是 README 收束时的事实与风格基线。
- 只要本次改动影响了用户可见能力、CLI 行为、配置方式、接入流程、runtime 导出、限制条件、使用示例或对外接口，AI agent 就应先更新 `PRE_README.MD`，而不是直接零散修改 README。
- README 应比 `PRE_README.MD` 更短、更稳、更主线化；不能脱离 `PRE_README.MD` 自行增删事实，也不能为了英文版“更自然”而改变信息密度、删段或补充另一套口径。
- 根 `README.md` 与根 `README_EN.md` 必须严格镜像：
- 章节顺序一致
- 标题数量与层级一致
- 表格行列与字段一致
- 代码块数量、顺序和参数一致
- 提示项、限制说明、示例数量一致
- `packages/frida-analykit-agent/README.md` 与 `packages/frida-analykit-agent/README_EN.md` 也必须遵循同样的严格镜像规则。
- 根 README 的默认结构应优先服务普通用户，再进入高级/开发用户，最后再放调试、测试、发布和仓库结构；不要把主线 CLI 用法和高级 runtime 细节混写。
- agent 包 README 的默认结构应优先按“包定位 -> 安装 -> 能力总览表 -> 普通使用方式 -> 高级能力 -> 调试与非公共内容”组织。
- 当能力面较多、导入路径较多或存在轻入口/重 capability 区分时，应优先使用表格提升可读性；尤其是 agent 包 README 的 capability 概览，默认使用表格而不是散列 bullet。
- 根 README 的架构说明必须读取并嵌入 `docs/arch.mermaid` 的 Mermaid 源码；在架构未变时，该文件视为只读事实源，不允许为了更新 README 顺手改图。
- 不要把根 README 和 agent 包 README 当作同一份文档的不同副本机械同步；它们面向的读者不同，只能共享事实基线，不能共享发布面。
- 当用户要求进行整体项目的 README 收束或 release 文档收束时，应基于 `PRE_README.MD` 先更新根 `README.md`，再翻译根 `README_EN.md`，再更新包 README，最后翻译包 README_EN。
- 中英文 README 更新完成后，必须做一次二次校对，至少逐项检查：
- 标题树是否完全一致
- 表格行列和字段是否完全一致
- 代码块数量、顺序和参数是否完全一致
- 提示项、限制说明和示例数量是否完全一致
- Mermaid 架构图是否与 `docs/arch.mermaid` 完全一致
- 如果某项能力已经实现，但尚未写入 `PRE_README.MD`，则视为文档维护未完成；如果已经写入 `PRE_README.MD`，但尚未收束到对应 README，则说明还处于中间态，而不是最终对外态。
- 如果中英 README 任一侧存在漏段、漏表、漏示例、漏限制说明，视为 README 收束未完成。

## 发版流程规则

- AI agent 不应在用户只说“发版一下”时自行猜测目标版本号；只有当用户明确给出要发布的版本号后，才进入正式发版准备。
- 如果用户给出的目标版本号与当前线上已发布版本不构成合理递增关系，或与当前仓库版本状态明显不一致，AI agent 必须先向用户确认版本号是否说错，再继续。
- 用户明确提出发版请求后，正式版本切换、打 tag、dry-run、发布前检查之前，必须先基于 `PRE_README.MD` 收束文档。
- 该文档收束顺序应为：
- 先更新根 `README.md`
- 再同步更新 `README_EN.md` 作为根文档的严格英文翻译
- 再更新 `packages/frida-analykit-agent/README.md`
- 再同步更新 `packages/frida-analykit-agent/README_EN.md` 作为包文档的严格英文翻译
- 两组 README 都完成后，必须先做一次双语镜像二次校对
- 文档收束完成后，才进入 `docs/release-process.md` 中定义的 release-version、preflight、RC、stable、publish 流程。
- 如果四份 README 中任意一份仍缺段、缺表、缺示例、缺限制说明，或者中英结构不一致，则不允许进入正式发版流程。
- 发版时 `git commit`、`git tag`、`workflow_dispatch`、`push tag` 必须串行执行，不要并行；尤其不要把 `git commit` 和 `git tag` 并行跑，否则 tag 可能落在错误提交上。
- stable 的远程 dry-run / 正式发布依赖远端已存在可追溯的 RC tag；如果 RC tag 还没有 push 到远端，不要直接继续 stable dry-run。
- 发版过程中如果发现严重 bug，不要立刻边发版边修；应立即停止当前发版流程，明确告诉用户问题出在哪里、影响范围是什么、发布风险是否高。
- “严重 bug” 包括但不限于：
- 会导致产物无法安装或无法运行
- 会导致关键主流程失效
- 会导致 release 契约、版本元数据或发布物明显错误
- 会导致高风险兼容性回退或公开错误发布
- 如果只是很小的 bug，且当前仍处于 RC 阶段，可以在 RC 分支修复后继续验证和发布；必要时递增 `rc.N` 重新走 RC 流程。
- stable 提升阶段应尽量避免再混入功能性修复；如果在 stable 提升时发现问题严重到影响发布正确性，应停止 stable 发布，而不是临时补丁后直接继续。

## 结合本项目的落地建议

- Python 侧优先沿 `src/frida_analykit/` 的职责边界拆分。
- Python 侧新增或改动模块时，应同步关注类型注解是否完整，尤其是公开接口、配置对象、会话状态与消息结构。
- Frida agent / TypeScript 侧优先沿 `packages/frida-analykit-agent/src/` 的功能域拆分。
- 涉及用户可见功能、命令行行为、配置项、接入方式的改动时，应同步维护 `PRE_README.MD`。
- 测试侧优先沿 `tests/` 与 `tests/device/` 的层级继续保持职责分离。
- 新增模块时，优先复用现有目录语义，避免引入含义模糊的新层级。

## AI Agent 提交前自查

- 这次改动是猜测性修复，还是已确认修复？
- 如果是猜测性修复，它是否足够小、足够局部、足够容易回退？
- 这次修复是否引入了不必要的复杂度？
- 是否存在非显而易见的修复点，但还没有加注释？
