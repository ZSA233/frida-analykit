# 开发工作区

🌍 语言: 中文 | [English](README_EN.md)

Generated by `frida-analykit` `${version}`.

这份 README 会和 `config.toml` 一起生成到你的工作区里。根 README 只保留最小上手路径，这里负责补充工作区配置、常用命令和 capability 导入习惯。

## 快速开始

1. 安装依赖：

   ```sh
   npm install
   ```

   生成出来的 runtime 依赖直接来自 npmjs，不需要额外 `.npmrc` 或 token。

2. 先改 `config.toml` 和 `index.ts`，再让 Python CLI 触发构建和注入：

   ```sh
   frida-analykit doctor --config config.toml
   frida-analykit build --config config.toml
   frida-analykit attach --config config.toml --build --repl
   frida-analykit attach --config config.toml --watch --repl
   frida-analykit spawn --config config.toml --build
   ```

   如果你预期的顶层 `console.log(...)` 没出现，先看 `agent.stdout` / `agent.stderr` 指向的日志文件。ESM import 会早于 `index.ts` 后续语句执行，bootstrap 或 import 失败通常会先落到这里。

3. 需要管理远端 `frida-server` 时，用同一份 `config.toml`：

   ```sh
   frida-analykit doctor --config config.toml
   frida-analykit doctor fix --config config.toml
   frida-analykit server install --config config.toml
   frida-analykit server boot --config config.toml
   ```

   如果你要固定某个 `frida-server` 版本，可以在 `config.toml` 里写 `server.version`，或者在 `server install` 时显式传 `--version`。

## 工作区里有什么

- `README.md`：中文工作区说明，重点解释配置和命令。
- `README_EN.md`：英文镜像版本，结构和代码块与中文版保持一致。
- `config.toml`：当前 CLI 主线配置文件；模板会随能力增长继续扩展。
- `index.ts`：你的 agent 入口。
- `package.json`：依赖与 `build/watch` 脚本；默认会固定到与当前 CLI release 对应的 `@zsa233/frida-analykit-agent` 版本。
- `tsconfig.json`：当前脚手架默认的 TypeScript 编译配置。

## `config.toml` 配置说明

`config.toml` 是按特性持续增长的模板。你可以先从下面这些字段理解它的职责边界：

- `app`：目标包名。`spawn` 时必填；`attach` 时也可拿它自动解析 PID。
- `jsfile`：注入使用的 `_agent.js` 路径。只要你改了构建输出路径，就要同步更新这里和 `package.json` 里的 `build/watch` 脚本。
- `[server].host`：设备链路。支持 `usb`、`local`、`host:port`；模板默认的 `127.0.0.1:27042` 适合 adb forward 场景。
- `[server].path`：设备侧 `frida-server` 路径。
- `[server].device`：可选的设备序列号。多设备在线时建议显式填写，优先级高于 `ANDROID_SERIAL`。
- `[server].version`：可选的 `frida-server` 版本。`doctor fix` 和 `server install` 会优先参考它。
- `[agent].datadir`：宿主机侧导出根目录。Dex、ELF 和其他产物默认都会从这里继续展开。
- `[agent].stdout` / `[agent].stderr`：agent 顶层日志输出位置。启动早期错误、import 失败和未捕获异常，优先看这里。
- `[script.rpc].batch_max_bytes`：通用 RPC 批量上限，不只服务于 Dex，也会影响 ELF 等大二进制传输。
- `[script.repl].globals`：`--repl` 中懒注入的 seed handle 名字。默认包括 `Process`、`Module`、`Memory`、`Java`、`ObjC`、`Swift`。
- `[script.dextools].output_dir`：Dex dump 的宿主机默认输出目录。
- `[script.elftools].output_dir`：ELF dump 的宿主机默认输出目录。
- `[script.nettools].output_dir`：SSL key log 的宿主机输出根目录。

当前模板里的行内注释就是对这些字段的最短说明。后续如果模板因为新能力继续增长，优先先看 `config.toml` 本身，再看这里的解释。

## Capability 导入与运行时习惯

`@zsa233/frida-analykit-agent/rpc` 现在只负责安装最小 RPC / REPL runtime。较重能力按需显式导入，这样 `frida-compile` 只会打包你实际用到的部分。

```ts
import "@zsa233/frida-analykit-agent/rpc"
import { help } from "@zsa233/frida-analykit-agent/helper"

setImmediate(() => {
  console.log("pid =", Process.id)
  console.log("api level =", help.runtime.androidApiLevel())
})
```

当前常见的 capability entrypoint 包括 `/helper`、`/process`、`/jni`、`/ssl`、`/elf`、`/dex`、`/native/libssl` 和 `/native/libc`。包根 `@zsa233/frida-analykit-agent` 仍然是瘦入口，不应该把它当成“自动带全家桶”的导入面。

## 进阶说明

- CLI 会复用工作区里的 `npm run build` 和 `npm run watch`，所以高级工作流依然可以直接跑：

  ```sh
  npm run build
  npm run watch
  ```

- 脚手架默认不再额外安装 `frida-java-bridge`。只有当你的 workspace 需要直接 import 它时，才自己加依赖。
- 如果你在 REPL / dump / keylog 之外又引入了新的 runtime 能力，优先把对应输出目录写进 `config.toml`，不要把产物散落在临时路径里。
