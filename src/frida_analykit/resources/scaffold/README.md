# Dev Workspace

Generated by `frida-analykit` `${version}`.

## Quick Start

1. Install dependencies:

   `npm install`

   The generated runtime dependency comes from npmjs, so no `.npmrc` or extra token is required.

2. Edit `index.ts` and keep `config.toml` pointing at `_agent.js`.

   `@zsa233/frida-analykit-agent/rpc` now only installs the minimal RPC / REPL runtime.
   Import extra capabilities explicitly so `frida-compile` only bundles what your script uses.
   Current lightweight capability entrypoints include `/config`, `/bridges`, `/helper`, `/process`,
   `/jni`, `/ssl`, `/elf`, `/native/libssl`, and `/native/libc`.
   The package root `.` is now a slim entry that only exposes the light base surface.

3. Let the Python CLI trigger the compile flow:

   `frida-analykit build --config config.toml`

   `frida-analykit attach --config config.toml --build --repl`

   `frida-analykit attach --config config.toml --watch --repl`

   If `agent.stdout` / `agent.stderr` point at `./logs/outerr.log`, check that file first when an expected top-level `console.log(...)` does not appear. ESM imports run before later statements in `index.ts`, so bootstrap or import failures will show up there before your own logs.

4. Manage the remote `frida-server` from the same config when needed:

   `frida-analykit doctor --config config.toml`

   `frida-analykit server install --config config.toml`

   Set `server.version` in `config.toml` or pass `--version` to pin a specific `frida-server` build.

5. The npm scripts still exist for advanced workflows, and the CLI reuses them:

   `npm run build`

   `npm run watch`

   The generated scaffold no longer adds a direct `frida-java-bridge` dependency.
   Add it yourself only if your workspace needs to import it directly.
