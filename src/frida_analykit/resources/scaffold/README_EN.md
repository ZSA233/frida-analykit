## Quick Start

1. Install dependencies:

   ```sh
   npm install
   ```

   The generated runtime dependency comes directly from npmjs, so no extra `.npmrc` or token is required.

2. Edit `config.toml` and `index.ts`, then let the Python CLI drive build and injection:

   ```sh
   frida-analykit doctor --config config.toml
   frida-analykit build --config config.toml
   frida-analykit attach --config config.toml --build --repl
   frida-analykit attach --config config.toml --watch --repl
   frida-analykit spawn --config config.toml --build
   ```

   If an expected top-level `console.log(...)` does not appear, check the files configured by `agent.stdout` / `agent.stderr` first. ESM imports run before later statements in `index.ts`, so bootstrap or import failures usually land there first.

3. When you need to manage the remote `frida-server`, use the same `config.toml`:

   ```sh
   frida-analykit doctor --config config.toml
   frida-analykit doctor fix --config config.toml
   frida-analykit server install --config config.toml
   frida-analykit server boot --config config.toml
   ```

   If you need to pin a specific `frida-server` build, set `server.version` in `config.toml` or pass `--version` to `server install`.

## `config.toml` Configuration Guide

`config.toml` is a capability-driven template and will keep growing over time. These are the fields you usually need to understand first:

- `app`: target package name. Required for `spawn`; `attach` can also use it to resolve a pid.
- `jsfile`: the `_agent.js` path used for injection. If you change the build output path, update this field and the `build/watch` scripts in `package.json` together.
- `[server].host`: device transport. Supports `usb`, `local`, and `host:port`; the template default `127.0.0.1:27042` is suited for adb-forward flows.
- `[server].path`: device-side `frida-server` path.
- `[server].device`: optional device serial. Fill it explicitly when multiple devices are connected; it takes precedence over `ANDROID_SERIAL`.
- `[server].version`: optional `frida-server` version. `doctor fix` and `server install` consult it first.
- `[agent].datadir`: host-side export root. Dex, ELF, and other artifacts grow from here by default.
- `[agent].stdout` / `[agent].stderr`: top-level agent log paths. For early bootstrap errors, import failures, and uncaught exceptions, check these files first.
- `[script.rpc].batch_max_bytes`: global RPC batch limit. It is not Dex-only and also affects larger ELF or binary transfers.
- `[script.repl].globals`: lazily exposed seed handles in `--repl`. The defaults are `Process`, `Module`, `Memory`, `Java`, `ObjC`, and `Swift`.
- `[script.dextools].output_dir`: default host-side output directory for Dex dumps.
- `[script.elftools].output_dir`: default host-side output directory for ELF dumps.
- `[script.nettools].output_dir`: host-side output root for SSL key logs.

The inline comments in the template are the shortest explanation for each field. As the template grows with new features, check `config.toml` itself first, then come back here for the broader intent.

## Capability Imports And Runtime Habits

`@zsa233/frida-analykit-agent/rpc` now installs only the minimal RPC / REPL runtime. Import heavier capabilities explicitly so `frida-compile` only bundles what your script actually uses.

```ts
import "@zsa233/frida-analykit-agent/rpc"
import { help } from "@zsa233/frida-analykit-agent/helper"

setImmediate(() => {
  console.log("pid =", Process.id)
  console.log("api level =", help.runtime.androidApiLevel())
})
```

Common capability entrypoints today include `/helper`, `/process`, `/jni`, `/ssl`, `/elf`, `/dex`, `/native/libssl`, and `/native/libc`. The package root `@zsa233/frida-analykit-agent` is still intentionally slim and should not be treated as an implicit ‚Äúimport everything‚Äù surface.

## Advanced Notes

- The CLI reuses the workspace `npm run build` and `npm run watch`, so advanced workflows can still use:

  ```sh
  npm run build
  npm run watch
  ```

- The scaffold no longer installs `frida-java-bridge` directly. Add it yourself only when your workspace imports it explicitly.
- If you add new runtime capabilities beyond the default REPL / dump / keylog paths, prefer declaring the matching output directories in `config.toml` instead of scattering artifacts into ad-hoc temporary paths.
